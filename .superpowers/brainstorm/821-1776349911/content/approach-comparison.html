技术方案对比

三种实现路径，各有取舍

A

方案 A：纯原生（C++ / Win32 API）

做法：用 C++ 直接调用 Win32 API 构建浮动窗口、窗口关联、系统托盘等所有功能。UI 用原生控件或 Direct2D 自绘。

✅ 方案 A 优势

  • 性能最优，资源占用极低
  • 对窗口操作 API 调用最直接、最灵活
  • 单文件可执行，无需运行时依赖
  • 最适合做窗口层级管理（always-on-top、owner window 等）

❌ 方案 A 劣势

  • UI 开发效率低，自绘成本高
  • 跨平台到 macOS 基本要重写
  • 数据存储、网络同步等需要手动实现
B

方案 B：Tauri + Rust + Web 前端（推荐）

做法：Rust 后端处理窗口管理、系统 API 调用、数据持久化；Web 前端（React/Vue）负责 UI 渲染。Tauri 2.0 支持多窗口和原生 API 调用。

✅ 方案 B 优势

  • UI 开发效率高（HTML/CSS/JS）
  • Rust 后端性能好，系统调用能力强
  • Tauri 2.0 原生支持多窗口管理
  • 天然支持 Windows + macOS，跨平台成本最低
  • 社区活跃，插件生态丰富
  • 打包体积小（~3-5MB vs Electron 的 ~150MB）
  • 你的原生开发经验 + Rust 学习曲线平缓

❌ 方案 B 劣势

  • 需要学习 Rust（如果你还没用过）
  • 某些底层窗口操作可能需要写原生插件桥接
  • Tauri 对极端自定义窗口行为的支持不如纯原生灵活
C

方案 C：Electron + Node.js

做法：Electron 主进程管理窗口，渲染进程做 UI。通过 Node.js 的 ffi-napi 或 edge-js 调用 Win32 API 实现窗口关联。

✅ 方案 C 优势

  • 全 JavaScript 技术栈，生态最成熟
  • 多窗口管理 Electron 原生支持
  • 大量现成的 UI 组件库
  • 跨平台成熟度高

❌ 方案 C 劣势

  • 打包体积大（150MB+），对于一个小工具太重
  • 内存占用高（每个窗口一个进程）
  • 窗口层级管理需要原生模块，调用链较长
  • 性能不如 Rust/C++ 方案
